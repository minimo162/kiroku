**Version**: 1.0
**Date**: 2026-03-30
**Author**: 侑輝
**Status**: Draft

-----

## 1. エグゼクティブサマリー

Kiroku は、業務PCの画面操作を自動記録し、AIで業務手順書と時間分析レポートを生成するデスクトップアプリケーションである。

スクリーンショットの定期取得 → ローカルVLM（Vision-Language Model）による画面状態のテキスト記述 → M365 Copilot Excel による構造化分析、という2層アーキテクチャにより、**機密情報を社外に出さずに業務の棚卸しを自動化**する。

Tauri v2 + Svelte 5 + Rust で構築し、単一の .msi インストーラーでチーム展開可能な形態とする。

-----

## 2. 背景と課題

### 2.1 現状の課題

経理部門（連結財務グループ）では、約90社の子会社に対する連結決算業務を毎月実施している。業務は SAP GUI、Excel、Outlook、Teams など複数アプリケーションにまたがり、手順が属人化しやすい。

現在の業務棚卸し手法は以下の問題を抱えている:

- **手動記録の負荷**: 業務手順を自分で書き起こすのは時間がかかり、忙しい時期ほど後回しになる
- **記録の粒度が粗い**: 自己申告ベースだと「連結PKG作業 2時間」のような大雑把な記録になり、具体的な操作ステップが残らない
- **属人化の可視化が困難**: 誰がどの業務にどれだけ時間を使っているか、客観的なデータがない
- **引き継ぎ資料の作成コスト**: 異動・退職時の手順書作成が大きな負担になる

### 2.2 機会

- ローカルで動作する小型VLM（Qwen3.5-0.8B）の登場により、CPU-only環境でも画像理解が実用的に
- M365 Copilot Excel で Opus 4.6 / GPT が利用可能になり、社内承認済み環境で高品質なテキスト分析が可能に
- Tauri v2 の成熟により、Rust バックエンドの高性能と軽量な配布が両立

-----

## 3. プロダクトビジョン

> **業務を「やりながら」記録し、AIが「勝手に」手順書にまとめてくれる世界をつくる。**

ユーザーは普段通り業務をするだけ。Kiroku がバックグラウンドで画面を観察し、翌朝には業務の全体像と詳細手順が手元にある。

-----

## 4. ユーザーと利用シナリオ

### 4.1 プライマリユーザー

**経理部門の実務担当者**（連結財務グループ）

- SAP・Excel・Outlook を日常的に使用
- 業務PC: AMD Ryzen 5 PRO 6650U / 16GB RAM / GPU無し / Windows 11
- ITリテラシー: 中〜高（PowerShell・Python の基本操作は可能）
- M365 Copilot ライセンスあり

### 4.2 セカンダリユーザー

**経理部門のマネージャー**

- チームメンバーの業務配分を把握したい
- 業務改善・効率化の根拠データがほしい
- 異動時の引き継ぎを標準化したい

### 4.3 利用シナリオ

**シナリオ1: 月次決算業務の手順書自動生成**

1. 月次決算期間中（5営業日間）、Kiroku をバックグラウンドで動かす
1. 毎晩、VLMが画面状態をテキスト化
1. 決算終了後、5日分のCSVをExcel Copilot（Opus 4.6）に投入
1. 「連結PKGデータ収集」「内部取引消去」「UPI計算」等の業務別手順書が自動生成される
1. レビュー・修正して正式な手順書として保存

**シナリオ2: 日常業務の時間分析**

1. 2週間、通常業務中にKirokuを常時稼働
1. CSVをCopilotで分析し、業務カテゴリ別の時間配分を可視化
1. 「メール対応に1日2.5時間使っている」「SAP帳票出力の待ち時間が累計45分」等の知見を得る
1. 業務改善提案の根拠データとして活用

**シナリオ3: チーム展開と業務標準化**

1. .msi インストーラーでチームメンバーのPCにKirokuを配布
1. 各メンバーの業務記録を集約・比較
1. ベストプラクティスを特定し、標準手順書を策定

## 6. 非機能要件

### 6.1 パフォーマンス

|項目            |要件                         |
|--------------|---------------------------|
|スクショ取得        |1回あたり500ms以内（業務操作に影響しないこと） |
|差分検出          |1回あたり50ms以内                |
|メモリ使用量（記録中）   |200MB以下（VLM未ロード時）          |
|メモリ使用量（VLM推論中）|2GB以下                      |
|VLM推論速度       |1枚あたり40秒以内（CPU推論）          |
|ディスク使用量       |スクショ一時保存: 1日あたり最大500MB（削除前）|
|VLMモデルサイズ     |約1GB（Q8_0） + mmproj 約300MB |

### 6.2 セキュリティ

|項目          |要件                          |
|------------|----------------------------|
|データ送信       |画像データの外部送信を一切行わないこと         |
|ローカル完結      |VLM推論は100%ローカルで実行すること       |
|画像削除        |VLM処理完了後にスクショ画像を確実に削除するオプション|
|CSV出力       |テキスト記述のみを含み、画像データを含まないこと    |
|M365 Copilot|社内承認済み環境でのみテキストデータを処理       |

### 6.3 ユーザビリティ

|項目      |要件                        |
|--------|--------------------------|
|インストール  |.msiインストーラーでワンクリック（管理者権限不要、perUserインストール）|
|初回セットアップ|モデルダウンロード含め15分以内          |
|日常操作    |起動後は放置。手動操作は翌朝のCSVエクスポートのみ|
|UI言語    |日本語                       |

### 6.4 互換性

|項目      |要件                             |
|--------|-------------------------------|
|OS      |Windows 11（Windows 10は動作対象外でも可）|
|WebView2|Windows 11標準搭載のものを利用           |
|CPU     |AVX2対応（Ryzen 5 PRO 6650U で動作確認）|
|RAM     |16GB以上推奨                       |

## 7. アーキテクチャ概要

### 7.1 2層アーキテクチャ

```
┌─────────────────────────────────────────────────┐
│  第1層: Kiroku デスクトップアプリ（ローカル）       │
│  ┌───────────┐  ┌──────────┐  ┌──────────────┐  │
│  │ Svelte UI │←→│ Tauri IPC│←→│ Rust Backend │  │
│  └───────────┘  └──────────┘  └──────┬───────┘  │
│                                      │          │
│                    ┌─────────────────┼────┐     │
│                    │                 │    │     │
│               ┌────▼───┐  ┌────────▼┐ ┌─▼──┐  │
│               │ xcap   │  │ llama   │ │ csv│  │
│               │ capture│  │ .cpp    │ │ out│  │
│               └────┬───┘  └────┬────┘ └─┬──┘  │
│                    │           │         │     │
│               screenshots   VLM推論    CSV    │
│               (一時保存→削除)           ファイル │
└─────────────────────────────────────────┼───────┘
                                          │
                                     CSVを手動で読込
                                          │
┌─────────────────────────────────────────▼───────┐
│  第2層: M365 Copilot Excel（社内クラウド）         │
│  ┌────────────────────────────────────────────┐  │
│  │ Copilot Excel Agent (Opus 4.6 / GPT)      │  │
│  │ → 手順書生成 / 業務一覧 / 時間分析         │  │
│  └────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

### 7.2 データフロー

```
スクショ取得 (30秒間隔)
    ↓
差分検出 (dHash比較)
    ↓ 変化ありのみ
メタデータ取得 (ウィンドウタイトル, プロセス名)
    ↓
一時保存 (PNG + metadata)
    ↓
[夜間バッチ] VLM推論 (Qwen3.5-0.8B)
    ↓
テキスト記述CSV出力
    ↓
スクショ画像削除
    ↓
[翌朝手動] Excel Copilot で分析
    ↓
手順書 / 業務一覧 / 時間分析
```

## 9. VLM連携のフォールバック設計

### Plan A: llama-cpp-rs 直接バインディング（推奨）

- Rustから直接llama.cppを呼び出し、プロセス間通信なしで推論
- 最速・最軽量だが、マルチモーダル（mmproj）のRustバインディング対応状況に依存

### Plan B: llama-server サブプロセス + HTTP API

- Rustから `llama-server` をサブプロセスとして起動
- OpenAI互換API（localhost:8080）経由で画像をbase64送信
- llama.cppのVLM機能をそのまま利用でき、安定性が高い
- llama-serverの起動・停止・ヘルスチェックの管理が必要

開発初期は Plan B で動作確認し、llama-cpp-rs のVLM対応が安定した時点で Plan A に移行する。

## 10. 第2層: Copilot Excel 連携

### 10.1 入力

Kiroku が出力した CSV ファイル（UTF-8 BOM付き）

### 10.2 分析プロンプト（Opus 4.6 推奨）

```
このExcelには、業務PCの画面状態を30秒間隔で記録した時系列データがあります。
各行は:
- timestamp: 記録時刻
- app: アクティブアプリケーションのプロセス名
- window_title: ウィンドウタイトル
- description: AIが画面内容を記述したテキスト

以下を新しいシートに作成してください:

■シート「手順書」
連続する記述から一連の操作フローを識別し、業務ごとに番号付き手順として記述。
各手順に[開始時刻][終了時刻][使用アプリ]を付与。
同じ業務が複数回出現する場合はパターンを統合。

■シート「業務一覧」
業務名 / 業務カテゴリ / 実施回数 / 平均所要時間 / 使用アプリケーション / 備考

■シート「時間分析」
業務カテゴリ別の時間配分集計、時間帯別の業務パターン、
改善余地がありそうな作業へのコメント。
```

### 10.3 期待される成果物

- **手順書シート**: 番号付きステップの操作手順（業務別）
- **業務一覧シート**: 全業務のカタログ
- **時間分析シート**: カテゴリ別時間配分、改善提案

## 12. リスクと対策

|リスク                     |影響             |対策                                                  |
|------------------------|---------------|----------------------------------------------------|
|Qwen3.5-0.8Bの画面記述品質が不十分 |手順書の精度低下       |プロンプト最適化 / 2B以上のモデルへの切り替え / OCR併用のハイブリッド構成          |
|llama-cpp-rsのVLM対応が不安定  |Plan Aが使えない    |Plan B（llama-server HTTP API）で確実に動作                 |
|CPU推論が想定より遅い            |夜間バッチが朝までに終わらない|差分検出の閾値を上げて処理枚数を削減 / スレッド数調整 / 推論パラメータ(max_tokens)削減|
|スクショに機密情報が含まれる          |情報漏洩リスク        |全処理ローカル完結 / 画像即時削除 / CSVにはテキスト記述のみ                  |
|業務PCへのソフトウェアインストールに制限がある|展開できない         |perUserインストール（管理者権限不要）で対応済み / IT部門との事前調整       |
|Copilot Excelのトークン制限    |大量データの分析に制約    |日次CSV（1日分ずつ）で投入 / 要約済みデータに前処理                       |
|バックグラウンド動作がPC性能に影響      |業務に支障          |スクショ取得・差分検出は軽量（<50ms）/ VLM推論はバッチのみ                  |
|Tauri v2 の WiX テンプレート制約    |perUser MSI ビルド成功|**解決済み（タスク33）**。カスタム WiX テンプレートで `InstallScope="perUser"` + `LocalAppDataFolder` を使用。ICE38 (KeyPath を HKCU レジストリに変更) と ICE03 (Handlebars `{{#each}}` ループ内で `{{../manufacturer}}` 親スコープ参照) を修正。ビルド時は `target/release/wix/` キャッシュ削除が必要な場合あり|

## 15. UI/UX 改善提案

現在の実装レビューで特定した摩擦ポイントを解消するための改善案。優先度は全画面への影響度と経理担当者（非エンジニア）の日常利用を基準に設定。

### 15.1 全画面共通

#### UX-001: グローバルトースト通知システム
**優先度:** Must
**現状:** 各画面が `message = $state<string | null>` で個別にインライン表示。成功・エラーが同じ色・位置に出るため見逃しやすく、自動消去されない。
**改善:** Svelte の `Context API` でグローバルトースト管理を実装。成功（brass）・エラー（cinnabar）・情報（ink）の3種を区別し、成功は3秒、エラーは手動で閉じるまで表示。`+layout.svelte` にトースト表示コンポーネントを追加する。

#### UX-002: 日付範囲の整合性バリデーション
**優先度:** Should
**現状:** エクスポート・履歴の両画面で開始日＞終了日を設定してもエラーなし。
**改善:** 開始日・終了日フィールドの `onchange` で比較し、不正な場合は「開始日は終了日以前にしてください」とインライン警告を表示。エクスポート・実行ボタンは無効化。

### 15.2 ダッシュボード

#### UX-003: 記録停止の確認ダイアログ
**優先度:** Should
**現状:** 停止ボタンを押すと即座に記録が止まる。誤タップのリスクがある。
**改善:** 停止ボタン押下時のみ、`<dialog>` タグによるネイティブダイアログ（または簡易モーダル）を表示し「記録を停止しますか？」の確認を挟む。開始は即時実行で問題なし。

#### UX-004: エラー状態のリセット
**優先度:** Should
**現状:** ダッシュボードの「最新エラー」は一度表示されると手動でクリアできない。前回のエラーが現在も発生中か区別しづらい。
**改善:** 最新エラー表示の横に「✕ クリア」ボタンを追加。ボタン押下で `clear_last_error` コマンドを呼び出し、エラー表示を消去。アプリ再起動時は自動クリア。

### 15.3 設定画面

#### UX-005: リアルタイム入力バリデーション
**優先度:** Should
**現状:** VLMホスト・最大トークン・バッチ時刻などの入力フィールドに保存前の検証がなく、無効値で保存しても保存時エラーにもなりにくい。
**改善:** 各フィールドに `oninput` / `onblur` バリデーションを追加。
- VLMホスト: `host:port` 形式チェック
- 最大トークン: 64〜2048の範囲チェック
- バッチ時刻: HH:MM形式チェック
バリデーションエラーはフィールド直下に赤字で表示し、保存ボタンは全エラー解消まで無効化。

#### UX-006: 設定の Ctrl+S 保存ショートカット
**優先度:** Could
**現状:** 設定変更後は必ずマウスで「設定を保存」ボタンをクリックする必要がある。
**改善:** 設定画面内で `Ctrl+S` のキーイベントをキャプチャし、保存処理をトリガー。ショートカット発動時はトースト（UX-001）で「設定を保存しました」を表示。

### 15.4 記述プレビュー画面

#### UX-007: VLM 記述への初期化ボタン
**優先度:** Should
**現状:** 手動編集後、VLMが生成した元の記述に戻す手段がない（編集履歴で確認はできるが、コピペが必要）。
**改善:** 記述エディタの右上に「元の記述に戻す」ボタンを追加（履歴の最初のエントリが存在する場合のみ表示）。押下後に確認を挟まず即時ロールバックし、トーストで通知。

#### UX-008: 記述保存の Ctrl+S ショートカット
**優先度:** Could
**現状:** テキストエリアで編集後、保存は画面右上のボタンをクリックする必要がある。
**改善:** テキストエリアにフォーカスがある状態で `Ctrl+S` を押すと保存処理を実行。ショートカットのみ `e.preventDefault()` して OS のダイアログを抑制する。

### 15.5 エクスポート画面

#### UX-009: アプリ一括選択・解除ボタン
**優先度:** Should
**現状:** アプリフィルターのチェックボックスが多い場合（10件以上）、全選択・全解除に手間がかかる。
**改善:** アプリフィルターの見出し横に「全選択」「全解除」リンクボタン（small, brass-700）を追加。

#### UX-010: エクスポート進捗インジケーター
**優先度:** Could
**現状:** エクスポートボタンが「エクスポート中...」になるのみで、件数が多い場合に進行状況が分からない。
**改善:** バックエンドから `export-progress` イベント（処理済み件数/全件数）を Emit し、エクスポート画面にプログレスバーを表示。1000件未満では不要なので、件数が1000件以上の場合のみ表示する。

### 15.6 履歴・検索画面

#### UX-011: 検索キーワードのハイライト
**優先度:** Should
**現状:** 検索結果の `description` と `window_title` に一致テキストが含まれていても、どの部分がマッチしたか分からない。
**改善:** 検索結果リストの各カードで、`query` と一致する文字列を `<mark>` タグ（brass-200 背景）でハイライト。クライアントサイドで文字列置換を行い、バックエンドの変更は不要。

#### UX-012: フィルター一括リセットボタン
**優先度:** Should
**現状:** キーワード・日付範囲・アプリフィルターを個別にしかリセットできない。
**改善:** フィルターパネルの上部に「フィルターをリセット」ボタンを追加。押下で全フィルターを初期値（キーワード空、日付空、アプリ全選択）に戻し、再検索を実行。
